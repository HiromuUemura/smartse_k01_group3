# グループで確認・決定が必要な事項

OpenAI を使った予定抽出機能（試作）を追加するにあたり、グループで合意・確認が必要な点をまとめます。
コード中では `要グループ確認` というコメントで該当箇所を示しています（`grep -rn "要グループ確認" .`）。

## 1. AIプロバイダの方針
- 今回は **OpenAI（GPT-5 nano / mini）** で実装。READMEには元々「OpenAI API」と記載あり。
- **確認**: 本当にOpenAIで進めるか（料金・各自のアカウント・キー配布方法）。

## 2. モデルID・APIの仕様
- 既定モデルを `gpt-5-nano`、選択肢に `gpt-5-mini` を用意（`lib/openai.ts` の `ALLOWED_MODELS`）。
- **確認**: `gpt-5-nano` / `gpt-5-mini` の **正式なモデルID** をOpenAIのドキュメントで確認する。
- **確認**: GPT-5系で Chat Completions API のままで良いか（`max_completion_tokens` などのパラメータ要否、
  Responses API への移行要否）。画像入力（`image_url`）と Structured Outputs（`json_schema`）の対応も要検証。
- `samples/prints/` の画像で抽出精度を検証し、nano で不足なら mini に切り替える。

## 3. APIキーの扱い（セキュリティ）
- GUIから入力したキーは **サーバ側の httpOnly Cookie** に保存し、サーバからのみOpenAIを呼ぶ実装。
  ブラウザのJSからは読めない。
- **確認**: 運用形態。
  - 「各自が自分のキーを入力して使う」前提なら現状の実装でよい。
  - 共有サーバに他人のキーが残る形は避ける。サーバ共通キーにするなら環境変数 `OPENAI_API_KEY` に寄せる。
- `.env`（キー類）は `.gitignore` 済み。キーは絶対にコミットしない。

## 4. 抽出スキーマ
- `lib/types.ts` の `ScheduleCandidate` を抽出の出力フォーマットとして使用。
- **確認**: 抽出したい項目（持ち物・締切・場所など）に過不足がないか。

## 5. 画像の扱い
- 現状は画面からアップロードした画像をその場で抽出するだけ（サーバに保存しない）。
- **確認**: アップロード画像を保存するか（プライバシー観点では抽出後破棄が無難）。
